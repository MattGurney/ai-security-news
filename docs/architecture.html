AI Security News Architecture

This document describes the system from an AI and data-flow perspective: how Hacker News items become security candidates, how agents cooperate, where deterministic filtering hands off to classifier and analyst agents, and how proactive monitor state avoids reprocessing old stories.

Objective

The system proactively scans Hacker News for stories that may matter to security and engineering teams. It favours recall early in the pipeline, then uses analysis stages to turn raw news into concise intelligence. The goal is not just to label stories as security-related; it is to explain why they matter, who should care, and how urgently they should be treated.

What the System Produces

The final output is an IntelligenceItem. It answers the practical questions a security-aware reader would ask after seeing a noisy news feed item:

  • What happened? A short summary of the story.
  • Why is it security-relevant? The analyst's security angle.
  • Who should care? The affected audience, such as appsec, platform, dependency-management, or engineering teams.
  • How urgent is it? An alert level and confidence score.
  • Why did the pipeline pick it up? Matched deterministic signals and classifier routing decisions.

Data Objects

NewsItem

A normalized story from Hacker News with title, URL, author, score, comments URL, source, and publish time.

SecurityCandidate

A story selected by deterministic recall signals. It carries a relevance score, matched signals, and an explainable reason.

IntelligenceItem

The published output: a candidate plus analyst assessment, alert level, confidence, audience, and security angle.

Proactive Runtime

The CLI monitor mode wraps the LangGraph workflow with local state. It polls Hacker News on an interval, records seen story IDs in SQLite, sends only new stories into the AI pipeline, persists emitted intelligence, and appends JSONL records for simple inspection. 

monitor loop
  -> fetch Hacker News stories
  -> exclude IDs already in data/news-monitor.sqlite
  -> run LangGraph over new stories only
  -> persist seen stories and intelligence rows
  -> append output/intelligence.jsonl
  -> sleep and repeat

Agent Roles

  • Fetcher Agent: retrieves top Hacker News stories through the Firebase API.
  • Deterministic Candidate Filter: performs cheap, explainable, recall-oriented triage using security, AI, incident, package ecosystem, and vulnerability signals.
  • Weak Classifier Agent: classifies candidates as ignore, monitor, or analyze using a lightweight prompt, with deterministic fallback routing when no key is available.
  • Strong Analyst Agent: uses an OpenAI model through LangChain structured output for candidates routed to deep analysis, or deterministic fallback analysis when no key is available.
  • Publisher: emits observable intelligence items to the CLI for the current demo surface.

Signal Strategy

The first filter is intentionally not the final judge. It catches likely security and AI-security material with enough recall to supply downstream model stages. The classifier agent then performs cheap semantic routing, and the stronger analyst is reserved for candidates that deserve deeper security interpretation.

Current deterministic signals include:

supply chain compromise vulnerability zero day exploit breach incident malware package ecosystem authentication cryptography prompt injection AI security AI system agentic system privacy data exposure

Routing Strategy

The model path is split into triage and analysis. The weak classifier makes a routing decision for each candidate, and LangGraph conditional routing skips the strong analyst when no candidates need deep analysis. 

SecurityCandidate
  -> WeakClassifierAgent: ignore | monitor | analyze
  -> route in LangGraph
  -> StrongAnalystAgent for analyze items
  -> Publisher

Planned Evolution

Future improvements include fetching article body text, adding more news sources, promoting monitored candidates when related stories accumulate, and adding a notification channel for high-severity intelligence.

A strong next step is retrieval-augmented analysis. The system could embed previously seen stories, article bodies, and emitted intelligence items, then retrieve related incidents before the strong analyst runs. Those related stories would give the final agent context about whether a new item is part of a broader campaign, a repeated vulnerability pattern, a supply-chain trend, or a follow-up to an earlier incident. 

new SecurityCandidate
  -> retrieve related stories from vector index / SQLite metadata
  -> pass candidate + related context to StrongAnalystAgent
  -> produce trend-aware IntelligenceItem

Limitations

  • The system currently uses only Hacker News top stories.
  • The deterministic filter sees only title and URL, not article body text.
  • The CLI is the only publishing surface.
  • Persistence is local-only and intended for demo/runtime state, not multi-user operation.
